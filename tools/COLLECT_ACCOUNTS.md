# collect_accounts.sh

A companion script to the `manage_user` Ansible role. Run it as root on an existing Ubuntu host to snapshot the current user accounts and service accounts into the var files that the role expects. The output can be used to bootstrap Ansible management on a machine that was previously configured by hand, or to audit what is present before making changes.

---

## Requirements

- Bash 4+
- Must be run as root (reads `/etc/shadow`-adjacent data and `~root`-owned `authorized_keys` files)
- Target: Ubuntu 18.04 or later (any system with standard `/etc/passwd` and `/etc/sudoers.d/`)

---

## Usage

```bash
sudo tools/collect_accounts.sh [USERS_OUT] [SVC_OUT]
```

| Argument | Default | Description |
|---|---|---|
| `USERS_OUT` | `vars/users.yml` relative to the repo root | Output path for regular user accounts |
| `SVC_OUT` | `vars/service_accounts.yml` relative to the repo root | Output path for service accounts |

Run from the repository root:

```bash
sudo tools/collect_accounts.sh
```

The script resolves default output paths relative to its own location, so `vars/users.yml` and `vars/service_accounts.yml` always land in the correct place regardless of the working directory.

Custom output paths:

```bash
sudo tools/collect_accounts.sh /tmp/users.yml /tmp/service_accounts.yml
```

## Configuration

UID classification ranges can be overridden via environment variables:

| Variable | Default | Description |
|---|---|---|
| `USER_MIN_UID` | `1000` | Lowest UID treated as a regular user |
| `USER_MAX_UID` | `65533` | Highest UID treated as a regular user (excludes `nobody` at 65534) |
| `SVC_MIN_UID` | `100` | Lowest UID treated as a service account (skips static system accounts 0–99) |
| `SVC_MAX_UID` | `999` | Highest UID treated as a service account |

Example — narrow the service account range to only UIDs you explicitly assigned:

```bash
sudo SVC_MIN_UID=900 SVC_MAX_UID=999 ./collect_accounts.sh
```

---

## What the Script Collects

### Regular users (`USER_MIN_UID`–`USER_MAX_UID`)

| Field | Source | Notes |
|---|---|---|
| `name` | `/etc/passwd` | |
| `uid` | `/etc/passwd` | |
| `comment` | `/etc/passwd` GECOS field | Omitted if empty |
| `ssh_public_key` | `~/.ssh/authorized_keys` | First non-comment key. Left blank with a note if the file does not exist. |
| `sudo` | `/etc/sudoers.d/<name>` | Set to `true` only when a drop-in file is present (matching what the role creates) |
| `groups` | `id -nG` | Supplementary groups only — primary group and same-name group are excluded |
| `shell` | `/etc/passwd` | Omitted when `/bin/bash` (the role default) |
| `home` | `/etc/passwd` | Omitted when `/home/<name>` (the role default) |

### Service accounts (`SVC_MIN_UID`–`SVC_MAX_UID`)

The same fields are collected, except `ssh_public_key` is never written (service accounts have no interactive login). Shell is omitted when `/usr/sbin/nologin`.

## Output Format

The script writes directly to the two var files. The output is ready to load into the playbook via `vars_files` without modification, though you should review it before use.

**Example `vars/users.yml`:**

```yaml
---
# Generated by collect_accounts.sh on 2026-05-08T10:00:00Z
# Fill in any empty ssh_public_key values before running the playbook.
manage_user_accounts:
  - name: alice
    uid: 1001
    comment: "Alice Smith"
    ssh_public_key: "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAA... alice@laptop"
    sudo: true

  - name: bob
    uid: 1002
    comment: "Bob Jones"
    ssh_public_key: ""  # no authorized_keys found — add before running playbook
    groups:
      - docker
```

**Example `vars/service_accounts.yml`:**

```yaml
---
# Generated by collect_accounts.sh on 2026-05-08T10:00:00Z
# Service accounts with UID 100–999.
# Remove any entries that should not be managed by Ansible.
manage_user_service_accounts:
  - name: deploy
    uid: 900
    comment: "Deployment automation"
    groups:
      - www-data

  - name: monitoring
    uid: 901
    comment: "Metrics and alerting"
    home: /var/lib/monitoring
```

## After Running

1. **Review the output.** The service accounts file will include every account in the UID range, many of which are created by packages and should not be managed by Ansible. Delete entries you do not want the role to touch.

2. **Fill in missing SSH keys.** Any user whose `authorized_keys` file was absent or empty will have `ssh_public_key: ""`. The role will fail validation for these accounts unless the key is added or the entry is removed.

3. **Check the multiple-key warning.** If a user has more than one authorized key, the script captures only the first and emits an inline comment. Decide whether `ssh_key_exclusive: true` is appropriate before running the playbook, as that would remove the additional keys.

4. **Run the playbook** once you are satisfied the var files represent the desired state:

```bash
ansible-playbook -i inventory.ini playbook.yml --check   # dry run first
ansible-playbook -i inventory.ini playbook.yml
```

---

## Limitations

- **One SSH key per user.** The `manage_user` role manages a single authorized key. If a user has multiple keys, only the first is captured. The rest will be left in place unless `ssh_key_exclusive: true` is set, in which case they will be removed on the next playbook run.
- **sudoers detection is limited.** Only `/etc/sudoers.d/<name>` drop-in files are checked. Sudo grants configured directly in `/etc/sudoers` or via other drop-in filenames will not be detected.
- **GECOS field.** The comment field in `/etc/passwd` sometimes contains additional comma-separated values (room number, phone, etc.) beyond the full name. These are captured as-is and may need trimming.
